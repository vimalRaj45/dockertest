Module 1: Docker Fundamentals

Learn about images, containers, and how Docker eliminates dependency issues.

Why Use Docker?

Before containers, running software was fraught with version mismatch issues ("It works on my machine!"). Docker encapsulates your application code, runtime (JRE), libraries, and settings into a single, isolated package called an Image.

Docker Image

A read-only blueprint (like a class in Java) containing all dependencies and source code needed to run the app.

Docker Container

A runnable instance of an image (like an object instantiation in Java) that runs isolated in its own sandbox environment.

Docker Registry

A repository where images are uploaded (like Docker Hub, GitHub Packages) so cloud platforms like Render can pull and run them.

Module 2: Writing the Dockerfile

Examine the multi-stage Docker build recipe designed for Spring Boot.

Multi-Stage Build Blueprint

To keep our final image lightweight and secure, we use a Multi-stage build. In Stage 1, we pull a full Maven image to compile our code. In Stage 2, we copy only the compiled JAR file into a slim JRE runtime image.

Dockerfile
# STAGE 1: Build the Maven Project
FROM maven:3.9.6-eclipse-temurin-17 AS builder
WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests

# STAGE 2: Lightweight JRE Run Stage
FROM eclipse-temurin:17-jre-jammy
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

Directive Explanations:

  • FROM: Selects the base image. We build with a compiler-ready JDK and run with a slim runtime-only JRE.
  • WORKDIR: Sets the working directory inside the container's filesystem.
  • COPY: Transfers source files from your local system into the image filesystem.
  • RUN: Executes commands (like compile/package) inside the container during build time.
  • EXPOSE: Tells Docker which port the app listens on inside the container (8080).
  • ENTRYPOINT: Configures the container to run as an executable. Here, it launches our JAR.

Module 3: Local Orchestration with Compose

Running the application and database together in local environments.

Local Multi-container Setup

Instead of manually starting containers and linking their IP addresses, Docker Compose allows us to define multi-container architectures in a single compose.yaml file.

compose.yaml
services:
  # Service 1: Local Postgres Database
  postgres:
    image: 'postgres:15-alpine'
    environment:
      - 'POSTGRES_DB=mydatabase'
      - 'POSTGRES_USER=myuser'
      - 'POSTGRES_PASSWORD=secret'
    ports:
      - '5432:5432'

  # Service 2: Spring Boot Application
  app:
    build: .
    ports:
      - '8080:8080'
    environment:
      - SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/mydatabase
      - SPRING_DATASOURCE_USERNAME=myuser
      - SPRING_DATASOURCE_PASSWORD=secret
    depends_on:
      - postgres

Key Features of Compose:

  • Service Networking: Docker Compose creates a default shared network. Inside the container context, our Spring Boot app can reach PostgreSQL by referencing its service name: jdbc:postgresql://postgres:5432/mydatabase.
  • depends_on: Ensures the database container starts up before the Spring Boot application starts trying to boot up.
  • Port Mapping (host:container): Maps port 8080 of our machine to port 8080 inside the container, allowing access via http://localhost:8080.

Module 4: Deploying to Render Cloud

Steps to deploy your Spring Boot app and database to Render for free.

Deploying Live Services on Render

Render supports hosting Docker containers directly. We can deploy our Spring Boot application and connect it to a managed database (or the Neon database you provided!).

1
Push Project to GitHub

Create a GitHub repository and push your project directory containing the Dockerfile, pom.xml, and source code. (Make sure the files are at the repository root or configure the sub-directory path in Render).

2
Create a Database

You can use your Neon Postgres Database URL: postgresql://neondb_owner:npg_UiY5nTrAclD2@ep-fragrant-field-ap8wb8z4-pooler.c-7.us-east-1.aws.neon.tech/neondb?sslmode=require. The application is pre-configured to connect to it automatically!

3
Create Web Service on Render

Log in to Render Dashboard. Click New + and select Web Service. Connect your GitHub repository.

4
Configure Settings

Configure the following settings in the Render Web Service creation panel:

  • Runtime: Select Docker (Render will automatically detect your Dockerfile and build it!).
  • Root Directory: Set to dockertest (since your Maven code and Dockerfile live in the subdirectory of your workspace).
5
Environment Variables

Expand the Advanced section in Render and add the environment variables so Spring Boot connects to your database securely:

Key Value
SPRING_DATASOURCE_URL jdbc:postgresql://ep-fragrant-field-ap8wb8z4-pooler.c-7.us-east-1.aws.neon.tech/neondb?sslmode=require
SPRING_DATASOURCE_USERNAME neondb_owner
SPRING_DATASOURCE_PASSWORD npg_UiY5nTrAclD2
6
Deploy and Monitor

Click Create Web Service. Render will pull your repository, execute the multi-stage Docker build, launch the container, and assign it a public URL (e.g. https://your-app.onrender.com)! Check the logs tab in Render to see Spring Boot startup logs.
